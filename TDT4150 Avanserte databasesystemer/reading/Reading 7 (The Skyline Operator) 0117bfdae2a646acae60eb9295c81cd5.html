Reading 7 (The Skyline Operator)

This article discusses the Skyline operation, which is the process of finding interesting data that meet specific criteria. The Skyline is defined as those points which are not dominated by any other point. The article explains how the Skyline operation can be useful for various applications, such as database visualization and finding good salespersons. The article also describes possible SQL extensions to specify Skyline queries and evaluates alternative algorithms to compute the Skyline. It also explains how standard index structures such as B-trees and R-trees can be exploited to evaluate Skyline queries and how the Skyline operation can interact with other query operations, such as joins. Finally, the article discusses related work, gives conclusions, and makes suggestions for future work.

We propose extending SQL's SELECT statement with an optional SKYLINE OF clause to specify Skyline queries. The dimensions of the Skyline are denoted by dl, ..., dm, and can be minimized, maximized, or different with MIN, MAX, and DIFF annotations. The SKYLINE OF clause selects all interesting tuples, i.e., tuples which are not dominated by any other tuple. If two tuples have the same values for all attributes listed in the SKYLINE OF clause and they are not dominated by any other tuple, then they are both part of the result. The semantics of the SKYLINE OF clause are straightforward and it is executed after the SELECT ... FROM ... WHERE ... GROUP BY ... HAVING ... part of the query, but before the ORDER BY clause and possibly other clauses that follow. A one-dimensional Skyline is equivalent to a min, max, or distinct SQL query without a SKYLINE OF clause. Transitivity is an important property to implement the Skyline operation.

We extend a database system with a new operator, the Skyline operator, to implement Skyline queries. This operator is executed after scan, join, and group-by operators, and before a final sort operator, if the query has an ORDER BY clause. Only simple modifications to the parser and query optimizer are required, making integration into a traditional SQL query processor very simple. Seven variants of the Skyline operator are described, based on block-nested-loops and divide-and-conquer algorithms, as well as one special variant for two-dimensional Skylines. Skyline queries can be implemented on top of a relational database system without changing the system, but this approach performs poorly.

To implement Skyline queries on a relational database system, we can translate the query into a nested SQL query. However, this approach has poor performance for several reasons, including being unable to unnest the query and having to execute joins or group-bys in both the outer query and subquery. To address these issues, we propose extending the database system by integrating a Skyline operator. This will allow us to combine the Skyline operation with other operations, such as join, resulting in little additional cost to compute the Skyline.

For one-dimensional Skylines, computing the Skyline is equivalent to computing min, max, or distinct. For two-dimensional Skylines, sorting the data based on the two attributes of the SKYLINE OF clause can be used to compute the Skyline. The test of whether a tuple is part of the Skyline is cheap because it only needs to be compared with its predecessor. However, sorting does not work for Skylines with more than two dimensions because there might be many tuples between a tuple and its predecessor that dominate it. Special algorithms exist to deal with three-dimensional Skylines, but they will not be discussed here for brevity.

The block-nested-loops algorithm reads tuples repeatedly and keeps a window of incomparable tuples in memory. When a new tuple is read, it is either eliminated, inserted into the window, or put into a temporary file for later processing. At the end of each iteration, tuples in the window that are not dominated by tuples in the temporary file can be outputted as part of the Skyline. The algorithm works well if the Skyline is small, but in the worst case, its complexity is O(n2). Despite having the same asymptotic complexity as the naive nestedloops algorithm, block-nested-loops performs better in terms of disk usage.

One way to speed up comparisons in the basic algorithm for skyline queries is to maintain the window as a self-organizing list. When a tuple in the window dominates another tuple, it is moved to the beginning of the window so that the next tuple from the input will be compared to it first. This approach is particularly useful for skewed data with a few dominant tuples and a significant number of neutral tuples in the skyline.

Another variant is to replace tuples in the window with a more dominant set. For example, if the window contains two hotels and the next hotel to be considered is incomparable to both of them, a replacement policy can be used to determine which tuples should be kept in the window. This approach requires additional CPU costs to determine the replacement victim and may result in some tuples being compared twice.

In this work, the authors propose a replacement policy based on the area covered by a tuple to decide which tuples should be kept in the window. However, the tradeoffs of this variant are not discussed in detail.

The divide-and-conquer algorithm (proposed in [9, 121]) is known to be the best algorithm in the worst case, but unfortunately, its complexity in the best case is also the same as the worst case. The algorithm is used to compute the Skyline, and it works by computing the median of the input tuples for a specific dimension and dividing the input into two partitions. The Skyline of each partition is computed recursively until the partition contains only one (or very few) tuples. Finally, the Skyline of the overall input is computed by merging the Skyline of each partition. The main challenge of the algorithm is in Step 3, where the trick is to partition the Skyline of each partition using an (approximate) median for a different dimension. This allows us to merge the Skyline of each partition efficiently.

M-way partitioning improves the basic algorithm's performance when input doesn't fit in main memory. Instead of dividing based on median, a-quantiles are used to ensure each partition fits in memory. M-way partitioning is used in the first and third steps, producing partitions that can be merged pairwise in main memory. A bushy merge approach minimizes the volume of data that must be read and rewritten to disk. An "Early Skyline" extension is proposed to eliminate dominated tuples in the first step, which reduces the number of tuples needing to be written and reread in partitioning steps. Early Skyline is attractive for selective Skyline results but incurs additional CPU cost.

Indices can be used to compute a Skyline, such as R-trees for nearest neighbor-like pruning or one-dimensional ordered indices like B-trees. By scanning through multiple indices simultaneously, a superset of the Skyline can be found and then pruned using algorithms from Section 3 to determine which tuples are part of the Skyline. This approach can be extended to Skylines with multiple dimensions and is useful for combined Skyline and Top N queries. Future work will explore these techniques in more detail.

The Skyline operator can be applied before a join or group-by if they are non-reductive, as it reduces the result size and makes the join cheaper. Additionally, pushing the Skyline operator through a join might make it possible to use an index to compute the Skyline. For example, in the query "SELECT * FROM Emp e, Dept d WHERE e.dno = d.dno SKYLINE OF e.age MIN, e.salary MAX," the Skyline of employees can be computed before the join. Optimizing for this push-down is similar to optimizing for Top N queries.

The query wants high-salary employees in a low-budget department. To optimize, we can use an "Early Skyline" approach before the join to eliminate some employees. We sort the employee table by dno and salary in descending order, then sort the department table by dno and apply a merge join. We can then compute the Skyline using any approach from Section 3. This approach reduces the number of employees considered and has no additional overhead if a sort-merge join is used.

Experiments were conducted on a Sun Ultra Workstation with 333 MHz processor, 128 MB memory, and Solaris 7 OS. All algorithms from Section 3 were implemented in C++ and run on benchmark databases stored on a 9GB Seagate disk. Variants included sort with "Early Skyline", BNL-basic, BNL-sol, BNL-solrep, D&C-basic, D&C-mpt, and D&C-mptesk. Skyline queries were also run on a commercial relational database system. Results are presented for three types of databases: indep (attribute values generated independently), corr (correlated attributes), and anti (anti-correlated attributes). The size of the Skyline for each database type is shown in Figure 12, and the RDBMS results are not directly comparable due to transaction processing. No indices were used, and the Skyline queries looked for tuples with low values in all dimensions.

The experiment compared the running times and disk VO for different algorithms used to compute two-dimensional Skylines for different types of databases. The results showed that the BNL algorithm was the clear winner for all three databases, even outperforming the special Sort algorithm that is only applicable for d=2. The D&C-mptesk algorithm was also effective, but the cost of applying an "Early Skyline" using Kung et al.'s algorithm was more expensive than sorting or carrying out nested loops. The other D&C variants that did not apply an "Early Skyline" showed poor performance due to high VO demands, and the basic D&C variant was particularly bad. The RDBMS also showed poor performance, especially for the anti-correlated database, where more tuples needed to be probed in the inner loop before elimination. However, the amount of disk VO carried out by the RDBMS could not be determined.

Our work builds upon the maximum vector problem algorithm by Kung et al. [9], extending it for better performance in a database context. We introduce the block-nested-loops algorithm as a new alternative, which performs significantly better in certain cases. While other algorithms for the maximum vector problem have been proposed for parallel computing or specific scenarios, they have not been applied in a database context. Related problems, such as the contour problem or computing the convex hull, have been studied in theory, but not in a database context. Nearest neighbor search and Top N query processing are also related to our work, but are different problems. We believe that the Skyline operator is essential for many applications, such as visualizing the Skyline of Manhattan, and can provide a "big picture" view before applying a Top N query for more specific results.

We extended a database system to compute the Skyline of a set of points using the SKYLINE OF clause in SQL's SELECT statement, proposed and evaluated alternative algorithms, discussed index usage, and analyzed the interaction with other query operators. The Skyline operation is useful for decision support and visualization. Our experimental results suggest implementing a block-nested-loops algorithm for good cases and a divide-and-conquer algorithm for tough cases. Future work includes exploring statistics for Skyline estimation, parallel computing, and approximate and online Skyline algorithms.
