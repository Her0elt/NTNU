Reading 13 (NOSQL Databases and Big Data Storage Systems)

The rise of large-scale applications storing massive amounts of data has led to the development of NoSQL databases as an alternative to traditional SQL systems. NoSQL databases are more flexible in their data modeling and do not require schemas, making them better suited for applications such as free email services and social media platforms with millions of users and diverse data needs. NoSQL databases are often used in combination with other data storage systems to accommodate the various types of data needed for these applications.

The characteristics of NoSQL databases are divided into two categories - those related to distributed databases and systems, and those related to data models and query languages. NoSQL systems are designed to be highly available, scalable, and performant, which is achieved through techniques such as replication, sharding, and high-performance data access. NoSQL systems often use eventual consistency rather than serializable consistency, and employ different replication models such as master-slave and master-master. Additionally, sharding is used to distribute the load of accessing large files or collections of data objects, and to improve load balancing and data availability. Finally, NoSQL systems typically use hashing or range partitioning on object keys to locate individual records or objects.

NOSQL systems have four main categories: document-based, key-value stores, column-based or wide column, and graph-based systems. It also lists additional categories such as hybrid NOSQL systems, object databases, and XML databases. The article explains that even keyword-based search engines can be considered as NOSQL big data stores.

The CAP theorem is introduced to explain the competing requirements of consistency, availability, and partition tolerance in a distributed system with data replication, and how it is not possible to guarantee all three properties at the same time. The article also explains that in NOSQL distributed data stores, weaker consistency levels are often acceptable, and guaranteeing availability and partition tolerance is more important than serializability. The article then introduces the four main categories of NOSQL systems: document-based, key-value stores, column-based, and graph-based. Each category is briefly discussed, and a representative system is introduced for each category.

Document-based NoSQL systems store data as collections of similar documents and are sometimes known as document stores. Unlike object and object-relational systems and XML, there is no need to specify a schema as the documents are self-describing data. Documents can have different data elements, and the system extracts the data element names from the self-describing documents in the collection. Users can request the system to create indexes on some of the data elements. MongoDB and CouchDB are examples of document-based NoSQL systems, and they can use different models, languages, and implementation methods.

MongoDB stores documents in BSON format and collections. Each collection has a unique ObjectId field called _id that is indexed automatically. The structure of the data fields in documents is chosen based on how documents will be accessed and used. There are three design options for interdocument references: denormalized, embedded with separate collection, or normalized. The choice of design depends on how the data will be accessed. Design tradeoffs discussed in other chapters of the book are applicable for choosing the appropriate design for document structures and document collections.

MongoDB offers CRUD operations, which are Create, Read, Update, and Delete. The insert operation is used to create and insert documents into collections, and the remove operation is used to delete documents based on a Boolean condition. The update operation can be used to modify or replace existing documents, and the find operation is used for read queries. The query condition can be specified as a Boolean condition, and the selected documents are returned as a result. For more detailed information, the MongoDB online documentation can be referenced.

MongoDB's atomic updates are for a single document, and multidocument transactions use the two-phase commit method to ensure atomicity and consistency. MongoDB uses the replica set concept to create multiple copies of the same data set on different nodes, and replication is done using a variation of the master-slave approach. Sharding, also known as horizontal partitioning, divides the documents into disjoint partitions known as shards to achieve load balancing. Range and hash partitioning are two ways to partition a collection into shards, and MongoDB queries are submitted to a module called the query router, which keeps track of which nodes contain which shards based on the partitioning method used.

Key-value stores are a type of distributed storage system that prioritize performance, availability, and scalability. The data model is simple, with a unique key used to locate data items quickly, and values that can be structured or unstructured. Retrieving values by supplying the key must be fast. There are many key-value store systems available, each with their own characteristics.

DynamoDB is a key-value store system offered by Amazon as part of its cloud computing services. It uses tables, items, and attributes to store data, where each item is a self-describing record containing attribute-value pairs. The primary key is used to locate items in the table and can be a single attribute or a pair of attributes. The system supports JSON format for items and hash indexes for fast data retrieval. The distributed characteristics of DynamoDB are similar to those of the open source key-value system Voldemort, which is based on similar techniques as DynamoDB.

Voldemort is a high-performance open-source system based on Amazon's DynamoDB, designed for horizontal scalability, replication, and sharding. Consistent hashing is used to distribute key-value pairs among the nodes of a distributed cluster. The system has simple basic operations and supports high-level formatted data values. The values can be specified in JSON or other data object formats with the help of the Serializer class provided by the user. The consistent hashing method enables horizontal scalability by adding new nodes to the system, and the sharding is built into the method. Replication is achieved by placing specified replicas of an item on successive nodes in a clockwise direction. When a node fails, its data items are distributed to other existing nodes whose labels follow the labels of the failed node. Voldemort has been used by LinkedIn for data storage.

Column-based or wide column are another category of NOSQL systems. BigTable, used in many Google applications, is a well-known example of this category. It uses the Google File System (GFS) for data storage and distribution, and Apache Hbase is an open source system similar to BigTable that typically uses HDFS for data storage. Hbase can also use Amazon’s Simple Storage System (S3) for data storage. In column-based systems, the key is multidimensional and has several components, including a combination of table name, row key, column, and timestamp. Column-based systems are different from key-value stores in terms of the nature of the key.

Hbase is a NoSQL system that organizes data using namespaces, tables, column families, column qualifiers, columns, rows, and data cells. Data is stored in self-describing rows with a unique row key, and each table is associated with one or more column families. Hbase stores multiple versions of a data item with a timestamp associated with each version. The concept of column family is similar to vertical partitioning, and each column family of a table is stored in its own files using the HDFS file system. A cell holds a basic data item in Hbase, and a namespace is a collection of tables.

HBase provides low-level CRUD operations (create, read, update, delete) and only supports simple operations, leaving more complex operations like joins between rows in different tables to be implemented by application programs. The create operation creates a new table and specifies one or more column families, while the put operation is used for inserting new data or new versions of existing data items. The get operation retrieves data associated with a single row in a table, and the scan operation retrieves all rows.

HBase tables are divided into regions that hold a range of row keys and are assigned to region servers. Each region has stores for each column family. A master server monitors the region servers and splits tables into regions, assigning them to region servers. HBase uses Apache Zookeeper for managing naming, distribution, synchronization, coordination, and replication services, and Apache HDFS for distributed file services. Zookeeper can have several replicas on several nodes for availability and keeps data it needs in main memory. HBase's distributed system architecture and components are not discussed in detail but full documentation is available online.

Graph databases or graph-oriented NOSQL systems represent data as a collection of vertices (nodes) and edges. Both nodes and edges can be labeled to indicate the types of entities and relationships they represent, and it is generally possible to store data associated with both individual nodes and individual edges. Neo4j is a particular open-source Java-based graph database system that will be discussed in detail in later sections, including its data model, querying capabilities, and distributed systems.

Neo4js data model uses nodes and relationships to organize data. Nodes and relationships can have properties, and nodes can have labels, which group nodes into collections. Relationships are directed and have a start and end node, as well as a relationship type. In comparison to conventional graph theory, Neo4j uses labels instead of entities, and relationships are directed. The CREATE command is used to create nodes and relationships, and multiple labels can be used for a single node. Paths specify a traversal of part of the graph, and a schema is optional in Neo4j. The system creates a unique identifier for each node, and indexes can be created to retrieve nodes using their properties efficiently.

Neo4j uses a high-level query language called Cypher which can be used to create, modify, and find nodes and relationships in a graph database. Cypher queries are made up of clauses, and the language can specify complex queries and updates. Some of the main clauses in Cypher include CREATE, DELETE, MATCH, and ORDER BY. Cypher also supports aggregation and conditions through clauses like WITH and WHERE. The language can be used to return both property values and the nodes and relationships themselves, allowing for graph visualization using Neo4j's tool. The full Cypher language manual is available online.

Neo4j has two versions: the enterprise edition and the community edition. Both versions support the Neo4j graph data model and storage system, the Cypher graph query language, and other interfaces. The enterprise edition provides additional features, such as caching and clustering of data and locking, to enhance performance. Neo4j also has a graph visualization interface to display a subset of nodes and edges in a database graph. It supports master-slave replication and caching to improve performance, and maintains logical logs to recover from failures.
